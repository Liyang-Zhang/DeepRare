# yk-FERTA 演示前端落地方案（第一版）

本文档定义一套区别于研发 `console` 的演示前端方案，用于：

- 对 IT / 平台团队演示 yk-FERTA 的接入方式
- 对业务方演示诊断流程不是黑箱
- 为后续正式前端接入提供页面结构和 API 映射基线

本文档不讨论账号、权限、历史中心、平台监控等平台职责。

## 1. 产品定位

演示前端的目标不是“把所有调试能力可视化”，而是把以下三件事讲清楚：

1. 录入病例后，系统必须先进行 HPO 提取与人工确认
2. 诊断是一个可追踪的长期任务，而不是一次性黑箱输出
3. 最终结果是结构化临床诊断卡，并能回放中间 artifacts

因此，演示前端应与研发控制台区分：

- 面向业务语义，不暴露底层调试字段
- 以表单、步骤、卡片为主，不以 JSON 为主
- 强调“录入 -> HPO确认 -> 运行 -> 审阅”主链路
- 强调诊断与分子确认边界

## 2. 页面信息架构

第一版采用 `1 + 2` 结构：

1. 演示入口页
2. 病例录入与 HPO 确认页
3. 任务执行与结果页

说明：

- 演示入口页用于讲清产品边界、流程、接口和入口
- 病例录入与 HPO 确认页负责创建 case 并发起 task
- 任务执行与结果页负责 SSE 状态、最终诊断卡、artifact 回放

现有研发控制台继续保留，但不作为对外交付页。

## 3. 页面职责

### 3.1 演示入口页

目标：

- 让首次访问者在 30 秒内理解系统作用和流程
- 区分演示页与研发控制台
- 提供进入“病例录入页”和“任务结果页”的清晰入口

展示内容：

- 产品定位
- 三步工作流
- API 对接边界
- 临床语义边界
- 快捷入口按钮

不展示：

- 原始 JSON 调试输入
- 工具原始返回
- 研发内部术语堆叠

### 3.2 病例录入与 HPO 确认页

目标：

- 完成病例输入
- 调用 HPO 提取
- 人工确认/删除/补充 HPO
- 创建 case + task

页面结构：

1. 病例基础信息
2. 临床资料录入
3. HPO 提取结果
4. HPO 搜索与补充
5. 已确认 HPO
6. 发起任务

关键约束：

- 未确认 HPO 时，不允许进入诊断主流程
- 所有进入主流程的 HPO 必须来自“确认集合”

### 3.3 任务执行与结果页

目标：

- 展示任务阶段状态
- 展示最终结构化诊断卡
- 展示 artifacts 回放能力

页面结构：

1. 任务状态区
2. 最终结果区
3. 可追溯 artifact 区

任务状态区展示：

- task_id
- status
- 当前阶段
- 进度
- 当前说明

最终结果区展示：

- summary
- diagnosis_cards
- next_steps
- cautions

artifact 区展示：

- hpo
- phenotype_hints
- evidence
- similar_cases
- reviews
- final_report

## 4. 关键展示原则

### 4.1 HPO 人工确认必须可见

HPO 确认不是附属功能，而是主流程前置条件。演示前端必须清楚展示：

- 自动提取只是初稿
- 医生/操作者需要删除误提取项
- 医生/操作者可以补充缺失 HPO
- 只有确认后的 HPO 会被用于诊断

### 4.2 临床诊断与分子确认必须分开

结果页必须清楚区分三层信息：

1. 临床诊断支持
2. 疾病层面的基因/机制知识
3. 当前患者本人是否已有分子证据

没有患者本人分子结果时：

- 允许展示 `disease_genes`
- 允许展示 `molecular_mechanism`
- 不允许暗示“当前患者已由某基因导致”

### 4.3 证据必须可追溯

演示页必须能让 IT 和业务方看到：

- 结果来自哪些阶段
- 支持证据、反对证据、缺失证据如何区分
- 相似病例和知识检索不是同一种证据

### 4.4 中文为主

页面整体以中文为主，英文术语仅保留：

- HPO
- OMIM
- Orphanet
- gene symbols
- 文献标题等必要字段

## 5. API 映射

第一版页面仅依赖已公开 API：

- `POST /api/v1/hpo/extract`
- `GET /api/v1/hpo/search`
- `POST /api/v1/cases`
- `GET /api/v1/cases/{case_id}`
- `POST /api/v1/tasks`
- `GET /api/v1/tasks/{task_id}`
- `GET /api/v1/tasks/{task_id}/events`
- `GET /api/v1/tasks/{task_id}/artifacts`
- `GET /api/v1/tasks/{task_id}/artifacts/{artifact_type}`
- `GET /api/v1/tasks/{task_id}/result`

要求：

- 页面不读取后端内部对象
- 页面不依赖 Python 运行态变量
- 页面不耦合调试专用结构

## 7. 分阶段实施计划

### Phase 1：演示入口壳

目标：

- 增加一个新的演示入口页
- 明确区分“演示入口”和“研发控制台”
- 不改变现有工作流逻辑

交付：

- 新静态页
- 新路由
- 基本可访问性测试

### Phase 2：病例录入页收口

目标：

- 在现有 `case_workbench` 基础上去调试化
- 强化 HPO 确认为主步骤
- 收敛文案和交互顺序

交付：

- 更接近业务语义的输入页
- HPO 提取/确认体验优化

### Phase 3：结果页收口

目标：

- 在现有 `task_viewer` 基础上强化结果卡与 traceability
- 降低 artifact 生硬感

交付：

- 任务状态更清晰
- diagnosis_cards 更突出
- artifacts 更适合回放演示

### Phase 4：联调与验收

目标：

- 用公开 API 完成完整演示链路
- 验证与 IT 交付边界一致

交付：

- 演示脚本
- 页面联调记录
- 已知限制说明

## 8. 第一版验收标准

满足以下条件即可认为演示前端第一版可用于内部演示：

1. 首次访问者能理解系统主流程
2. 能看出 HPO 确认是必经步骤
3. 能通过页面完成 case + task 运行
4. 能实时看到阶段推进
5. 能看到结构化诊断卡
6. 能查看关键 artifacts
7. 能明确区分临床诊断与分子确认边界

## 9. 已知限制

第一版仍然是本地静态前端，不是最终产品 UI：

- 无账号权限
- 无历史中心
- 无多病例管理
- 无平台级审计与监控
- 无正式视觉系统接入

但它应足以支撑：

- 内部产品评审
- IT 接口演示
- 流程说明
- 字段与页面职责对齐
