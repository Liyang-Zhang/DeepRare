# yk-FERTA 部署与交接边界（2026-04-29）

本文档定义 yk-FERTA 当前阶段面向公司服务器部署和 IT 交接的边界。

## 1. 交付目标

本项目交付的是：

- 可容器化部署的智能体核心服务
- 提供病例接入、任务创建、SSE 进度、结果获取、追溯产物查询接口
- 提供运行所需的核心配置、数据依赖说明和接口契约

本项目当前**不交付**：

- 鉴权
- 网关接入
- 正式业务前端
- 平台级日志采集
- 平台级监控告警
- 外围审计与权限体系

这些外围能力默认由 IT / 平台侧负责。

## 2. 建议部署形态

当前建议部署形态为单服务 Docker 容器：

- 服务进程：`uvicorn yk_ferta.api.app:app`
- 持久化：本地 SQLite 文件 + 挂载数据目录
- 配置方式：环境变量 + 挂载配置文件

当前阶段不要求 Kubernetes；非 K8s 服务器部署是可接受的。

## 3. 运行依赖边界

### 3.1 容器内核心依赖

- Python 运行环境
- `src/yk_ferta` 主代码
- `api/` 与 `tools/` 中仍被主流程复用的共享组件
- `config/clinical_mvp.json` 或其部署等价配置

### 3.2 挂载/随镜像提供的数据依赖

- `database/` 下的疾病知识与病例库文件
- `data/yk_ferta.sqlite3` 所在目录

### 3.3 外部依赖

- RAG-HPO 服务
- HPO 官方 API
- PubMed 检索出口
- 通用 Web 检索出口
- 可选 phenotype 工具服务

这些依赖应在部署文档中标明：

- 是否强依赖
- 失败时是否允许降级

## 4. 当前推荐责任分工

### 4.1 yk-FERTA 侧负责

- 核心工作流逻辑
- API 契约冻结
- 结果与 artifact 结构
- 关键配置项说明
- Docker 化运行说明
- 核心错误码与降级语义

### 4.2 IT / 平台侧负责

- 反向代理 / 域名 / 端口暴露
- 鉴权与访问控制
- 容器编排与启停
- 统一日志采集
- 平台监控与告警
- 正式业务前端开发

## 5. 当前进入部署前还必须完成的事项

### P0

- 冻结 REST API / SSE / `/result` 契约
- 明确 artifact 的用途边界
- 输出 Docker 运行依赖说明

### P1

- 补充核心健康检查说明
- 明确外部依赖失败的降级策略
- 明确数据目录与配置文件注入方式

## 6. 交付原则

- 正式前端主消费 `/api/v1/tasks/{task_id}/result`
- `/artifacts` 只用于追溯与审计，不作为唯一业务展示数据源
- 私有历史检测案例只能作为 `testing_finding_reference`
- 没有患者本人分子证据时，不输出“已确认基因病因”
