# yk-FERTA 交付清单（2026-04-29）

本文档面向 yk-FERTA 核心服务交付，目标是明确：

- 哪些内容必须由当前团队交付
- 哪些内容由 IT / 平台侧接手
- 上线前双方需要共同确认什么

---

## 1. 交付目标

本次交付对象是：

- 可部署的 yk-FERTA 智能体核心服务
- 面向前端/平台可调用的 REST API + SSE 接口
- 可追溯的任务、结果和 artifact 输出
- Docker 化部署所需的最小运行说明

本次**不交付**：

- 正式业务前端
- 鉴权
- 平台级监控告警
- 网关接入
- 公司统一日志平台集成

---

## 2. yk-FERTA 团队必须交付

### 2.1 服务代码与接口

必须交付：

- `src/yk_ferta/` 主服务代码
- `api/`、`tools/` 中仍被主流程依赖的共享组件
- 已冻结的 REST API 契约
- 已冻结的 SSE 事件契约
- `/result` 结果结构定义
- `/artifacts` 追溯用途说明

完成标准：

- 对外接口字段稳定
- 示例请求/响应齐全
- 错误码和重试语义明确

### 2.2 配置与数据依赖说明

必须交付：

- 主配置文件说明
- 环境变量说明
- 数据目录说明
- 外部依赖地址配置说明
- 各外部依赖失败时的降级语义说明

完成标准：

- 新环境可按文档完成最小启动
- IT 不需要读源码才能知道如何注入配置

### 2.3 Docker 化运行说明

必须交付：

- Dockerfile 或等价构建说明
- 启动命令
- 挂载目录说明
- 端口说明
- 健康检查接口说明

完成标准：

- IT 能用文档把服务以单容器方式跑起来

### 2.4 结果语义边界

必须交付：

- 临床诊断与分子确认分离原则
- 私有历史检测案例仅作 `testing_finding_reference`
- `diagnosis_cards` 的正式字段字典
- `artifacts` 仅作追溯/审计用途说明

完成标准：

- 前端和平台不会把结果字段误解释成“已确诊分子病因”

### 2.5 基本回归验证

必须交付：

- 核心接口单元测试
- 关键工作流回归测试
- 代表性病例最小验证集

完成标准：

- 每次交付前能确认主链路未断

### 4.1 接入与安全

由 IT / 平台侧负责：

- 网关接入
- 域名与证书
- 内外网访问策略

### 4.2 平台运维能力

由 IT / 平台侧负责：

- 容器部署与启停
- 统一日志采集
- 平台监控与告警
- 资源配额
- 备份与恢复策略

### 4.3 正式前端

由 IT / 前端侧负责：

- 正式业务前端开发
- 页面路由与交互
- 账号体系接入
- 页面级异常提示

说明：

- 正式前端主消费 `/api/v1/tasks/{task_id}/result`
- `/artifacts` 只用于追溯展示，不作为唯一业务数据源

## 5. 上线前双方联调确认

### 5.1 接口联调

需共同确认：

- `POST /api/v1/cases`
- `POST /api/v1/tasks`
- `GET /api/v1/tasks/{task_id}`
- `GET /api/v1/tasks/{task_id}/events`
- `GET /api/v1/tasks/{task_id}/result`
- `GET /api/v1/tasks/{task_id}/artifacts`

确认点：

- 请求字段理解一致
- 错误码处理一致
- SSE 断线重连策略一致

### 5.2 运行依赖

需共同确认：

- 配置文件放置方式
- 数据目录挂载方式
- SQLite 文件位置
- 外部依赖连通性
- 服务器出网策略是否满足 PubMed / HPO API / 其他外部工具需要

### 5.3 结果语义

需共同确认：

- 前端如何展示 `support_level` / `confidence`
- 如何展示 `missing_evidence`
- 如何提示“建议检测”而非“已确诊”
- 如何区分公共病例参考与私有历史检测参考
